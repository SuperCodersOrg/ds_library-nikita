# Section 1 – Specific Bug

While reviewing yesterday's implementation, I identified a design issue rather than a compiler or runtime error.

The DynamicArray was implemented using `malloc()` and `realloc()`, which worked correctly for primitive data types but could not safely handle C++ objects such as `std::string`. The problem was that constructors and destructors were never called, making the implementation unsafe for generic object types.

This issue required redesigning the DynamicArray before continuing with the implementation.

---

# Section 2 – Failed Attempt

My first implementation relied on `malloc()` and `realloc()` for all memory management because they seemed sufficient after testing with primitive data types.

After understanding how C++ object lifetime works, I realized that `realloc()` cannot safely resize memory containing constructed objects since it simply copies raw bytes. Continuing with this implementation would cause undefined behaviour for classes that require constructors and destructors.

Instead of trying to patch the existing implementation, I decided to redesign the DynamicArray completely. I updated the design proposal to use `operator new` for memory allocation, placement `new` for object construction, explicit destructor calls for object destruction, and `operator delete` for releasing memory. After finalizing the revised design, I implemented the DynamicArray again using this new approach.

# Section 5 – Learning Reflection

Today's work completely changed my understanding of memory management in C++.

Yesterday I learned why my first implementation was incorrect, and today I applied that understanding by redesigning the entire DynamicArray instead of trying to fix small parts of it. I now understand that allocating memory and constructing an object are two different operations. Placement `new` allows objects to be constructed inside previously allocated memory, while explicit destructor calls ensure that object resources are released correctly before the memory is freed.

Another important lesson was that changing the design early is better than continuing with an incorrect implementation. Updating the proposal first made the implementation much clearer because every method followed the same memory management strategy. Writing multiple test cases for every function also gave me confidence that the new implementation works correctly and behaves consistently after the redesign.
